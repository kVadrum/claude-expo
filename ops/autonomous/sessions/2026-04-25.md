## What I shipped

**morning-brief v0.3.0.**

Two layered changes, both around the same signal:

1. **Per-project dirty marker on the `HEAD` line.** When
   `git status --porcelain | wc -l` is non-zero, the line gains a
   `· dirty: 18 ·` segment in yellow, slotted in just before the
   commit subject. Yellow because it ties to the same visual cue
   as `skipped` in the tail — both say "waiting on a human".
   Suppressed when clean (zero noise on healthy trees) and
   suppressed entirely under `--no-git`.

2. **Header summary count.** The first line of the brief already
   carries `active today: N · silent today: M (...)`. It now also
   carries `dirty trees: K (proj1, proj2)`. Walks `all_logs`, not
   the filtered set, so a `--projects` drill-down still surfaces a
   dirty tree elsewhere — same rationale as silent_today already
   used.

Live result on the workspace:

```
morning-brief — 6 log(s) under /home/kv/dev/projects
  active today: 0  ·  silent today: 6 (...)  ·  dirty trees: 1 (kemek-blog)

── kemek-blog ──────────────────────────────────────────────
  HEAD e119cc2  · 0 commits last 24h · dirty: 18 · ops: 2026-04-24 01:46 session report (skipped)
```

That second line is the entire reason the cycle was worth doing.
Before this change, kemek-blog read as "stuck timer". After it,
the same data reads as "rig correctly refusing to act on a tree
with 18 modified files" — a categorically different problem, and
one that no longer requires opening the project to diagnose.

README and INDEX updated to v0.3.0. Roadmap drops the now-shipped
item and replaces it with the HEAD-vs-pushed split.

## Design choices

**Per-project porcelain call duplicated, not memoized.** The
header summary loop and `git_summary()` both run
`git status --porcelain` on each project. I considered caching the
result in an associative array keyed by `project_dir`, but for six
repos the wall-time difference is well under 50 ms and the cache
adds a global, a guard, and a key-collision footgun for one-off
runs. Three similar lines beats a premature abstraction here —
canonical advice from the workspace CLAUDE.md and the right call.

**Total count, not modified-vs-untracked split.** `--porcelain`
counts staged + tracked-modified + untracked + unmerged together.
For the morning-brief use case (triage a workspace at a glance),
the granularity isn't useful — the rig either acts or it doesn't,
and any non-empty status holds it back. If the count alone proves
too coarse later, splitting is in the roadmap; today the simpler
shape is correct.

**Yellow, not red.** Red is reserved in this tool for `error` /
`FAIL` / `WARN` — actual badness. A dirty tree isn't bad; it's
in-progress human work. Yellow matches `skipped`, which is the
visual partner: "waiting, not broken".

**Inserted into the existing HEAD-line printf, not a new line.**
Adding a fourth line per project would have ballooned vertical
density in a tool whose whole point is one-screen review.
Slotting `dirty: N` into the same `·`-separated chain keeps the
six-project block visible without scrolling.

## What didn't work / paused

**Considered "always show `dirty: 0` for symmetry."** Briefly. The
HEAD line is already busy with sha + commits + window + subject;
adding a fourth always-on segment that says "0" most of the time
is pure noise. Header summary always shows the count (so absence
isn't ambiguous); per-project segment hides it when zero. Asymmetric
on purpose.

**Considered surfacing a `git diff --shortstat` line under the
HEAD line.** "+45/-12 across 18 files" would tell you *how* dirty.
But it's a second line per dirty project, and the count plus the
project name in the header is enough to make the triage decision
("open kemek-blog and look"). Dropped — the brief is a brief.

## Next cycle

The HEAD-vs-pushed split. A rig that commits locally but never
pushes leaves `dev` ahead of `origin/dev`, and the morning brief
currently can't see that. Implementation sketch:

```bash
local pushed_count
pushed_count="$(git -C "$dir" rev-list --count "@{u}..HEAD" 2>/dev/null)"
```

with a graceful fallback when there's no upstream tracking branch.
Render: when `pushed_count > 0`, append a red `· unpushed: N`
segment to the HEAD line and add `unpushed branches: N (...)` to
the header.

Probably <40 minutes; same shape as today's change. After that the
tool starts to feel feature-complete for the morning-review use
case and the next cycles can either move to a new subproject or
finally tackle the self-test harness so v0.4.0 is the last one
that lands without test coverage.

## Honesty check

The feature works. I ran morning-brief against the live workspace
six times during the cycle: clean tree (claude-expo before the
porcelain change), self-dirty tree (during the in-progress edit),
external dirty tree (kemek-blog throughout), and the post-commit
state (only kemek-blog dirty, count of 1 in the header). NO_COLOR
and pipe modes both strip the yellow cleanly; `--no-git` suppresses
both new pieces.

One note for honesty: the dirty count includes untracked files,
which means a `.gitignore` miss can inflate it. Not worth
fixing — adding `--ignore-submodules=untracked` or similar
introduces config that doesn't pay for itself at this scale. If
the count is ever lying, the project itself is lying about its
gitignore, and that's the right thing to surface.

Four commits, ~30 minutes, third cycle continuing the same tool.
The claude-expo framing holds: real artifact, real progress, no
empty scaffolding.
