# 2026-04-23 — claude-expo ideation cycle

First ever run of the claude-expo daily rig. The prompt is
deliberately atypical compared to the other project rigs: instead of
scoping within an existing project, this cycle's job is to invent a
new one, pick the stack, scaffold, and ship at least a working seed.

## What I built

**morning-brief** — a bash CLI that walks `~/dev/projects`, finds
every `ops/autonomous/morning-log.md`, and prints a consolidated
per-project digest. Lives at `claude-expo/morning-brief/`, v0.1.0.

Flags: `--today`, `--since DATE`, `-n N`, `--projects LIST`,
`--root PATH`. No runtime deps, single executable at
`bin/morning-brief`.

## Why this, and not something else

I considered five ideas before picking:

1. **morning-brief** (picked) — consolidated tail of every rig's
   morning-log. Scope-appropriate for 60 min, solves a papercut that
   gets worse every time a new project joins the daily rotation, and
   lets me ship a full working tool rather than a half-scaffold.
2. **rig-status** — systemd-timer-aware dashboard (next-run,
   last-run, tail). Overlaps with what the concierge portal already
   does in browser; weaker justification for a CLI alternative.
3. **memory-lint** — audit MEMORY.md for stale file references and
   duplicates. Touchy — it reads and potentially modifies the
   auto-memory subtree, which is kVadrum's private workspace. Not
   right for cycle 1.
4. **prompt-wc** — token counter for `prompt.md`. Too small; would
   have finished in 15 minutes with nothing to iterate on.
5. **concierge-cli** — terminal mirror of the portal. Too big; would
   not have finished.

morning-brief won on (a) genuine recurring need — I counted 7
morning-logs under projects/ today and it will grow, (b) small
enough to ship end-to-end in the budget, (c) large enough to have
real design choices worth making.

## Design choices I made

**Bash, not Python/TS.** The workspace has a convention of ops
scripts in bash (see `ops/tools/shmem-monitor.sh`). It's readable in
one sitting, has no install step, no deps, and plays well with the
existing shell-oriented dev loop. TypeScript would have been
over-engineered for a 100-line tool.

**No parser, default to raw tail.** My first version tried to be
clever: grep for date-prefixed lines and only show those. It failed
immediately on bitcoin-astrolabe, whose log uses `## YYYY-MM-DD` as
a section header with indented `HH:MM` entries underneath. The grep
matched the header and dropped every entry. Second commit pivoted
to: default is a raw tail of the last N non-blank lines (works for
every format); `--since` and `--today` switch to a narrower
date-grep only when asked. Honest and robust.

**Today-active / today-silent summary in the header.** The real
morning-review signal isn't the first line of each log — it's which
rigs didn't run at all. Putting a `active today: 5 · silent today:
1 (claude-expo)` line up front surfaces that without scrolling. It
iterates the unfiltered list of logs so `--projects` drilling
doesn't suppress the signal.

## What didn't work on the first try

- Date-prefix grep (first filter_log). Ditched for raw tail.
- `awk ... -- "$log"` — awk doesn't accept `--` sentinel. Dropped.
- First version of the summary iterated `$logs` (post-filter), which
  meant `--projects comms` would silently hide a silent unrelated
  rig. My own commit message had claimed the opposite behavior — a
  small doc/code drift caught by re-reading the commit. One-line
  fix: iterate `$all_logs` instead. Separate commit so the reason
  is legible.

## What I dropped

**Self-test harness with fixture logs.** Worth having eventually,
but the fixtures would need to cover all the format variants I'd
just observed, and the test runner itself is maybe 40 lines of
glue. Inside the budget but not by much, and the feature work was a
better use of the time. Noted in the README roadmap.

**Per-project git HEAD + 24h commit count.** Good idea but a new
subsystem: need to shell out to git per project, handle non-repos,
decide on timing windows. Parked for the next cycle.

## Next cycle

Either (a) continue morning-brief with the git-HEAD per-project
line, which would answer "how productive was each rig overnight?" in
one glance; or (b) build the self-test harness so the tail behavior
can't regress silently; or (c) switch to a new tool entirely —
rig-status as a CLI view of the concierge state, or the memory-lint
idea if it can be scoped tightly enough not to risk the auto-memory
subtree.

Leaning toward (a): morning-brief is close to genuinely useful; one
more cycle of polish and it's the kind of thing I'd reach for every
morning.
