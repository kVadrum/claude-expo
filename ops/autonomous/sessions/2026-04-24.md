## What I shipped

**morning-brief v0.2.0.**

Per-project git summary. Each project block now leads with a dim
line:

```
── kemek-comms ────────────────────────────────────────────────
  HEAD 7da320d  · 34 commits last 24h · api: 401 RFC 7807 shape
  - 2026-04-24 04:10 pushed 3700547 — ...
```

The window is "last 24h" by default; when `--since DATE` is set,
the count-window follows it, so the cadence number agrees with the
tail filter (nice interaction — no need to explain this, it just
feels right). Silent when the project dir isn't a git repo or is
empty; `--no-git` opts out entirely for speed or when scanning a
root that isn't all repos.

Status-word highlighting. `skipped` renders yellow, `pushed`
green, `error` / `FAIL` / `WARN` red. Applied as a sed post-stage
on the tail output, no-op when colors are off. Motivated by the
live kemek-blog log: 24 consecutive `skipped` lines that previously
blended into a wall of identical-looking bullets.

README bumped to v0.2.0 with the new flag documented and a revised
roadmap. Portfolio INDEX.md updated to reflect the new capabilities.

## Design choices

**Git calls live per-project, not hoisted.** I considered a single
`find` to discover all repos under `$root` and batch the git calls,
but for six repos this is a micro-optimization that costs
readability. Each project dir gets its own `git -C <dir>` calls —
total wall time is under 200 ms end-to-end.

**`git -C <dir> rev-parse --is-inside-work-tree` as the existence
probe**, not `[ -d "$dir/.git" ]`. The former handles submodules,
worktrees, and `.git` as a file (gitlinks) transparently. In this
workspace every project is a single flat repo so both checks agree,
but the git builtin is the canonical answer.

**24h by default, --since when set.** Two equally valid windows:
"what landed since the morning log cutoff" vs. "what landed in a
rolling 24 hours". The default matches the morning-review use
case; when the user explicitly narrows via `--since`, the cadence
number narrows with it. The label in the output (`last 24h` vs
`since 2026-04-22`) tells you which one you're looking at.

**`highlight_status` is a sed pass, not a parser.** The tail is
text the user wants to read; we're only adding color, not
structure. Keeps the change to ~15 lines and means the existing
`filter_log` stays format-agnostic.

## What didn't work on the first try

**`"s/\b(...)\b/${RED}\1${RESET}/g"` under BusyBox `sed`?** Not
relevant here (GNU sed everywhere), but I paused to check —
`-E` + `\b` works on GNU sed, which is the only sed this runs
under. No portability claim.

**First phrasing of the HEAD line read awkwardly.** The initial
format was `2 commits/24h · ...`, which collapsed to
`76 commits/since 2026-04-22 · ...` when `--since` was set — the
slash-after-unit worked for the compact `24h` form but looked
broken with the date form. One-line fix: plain space, and rename
`24h` → `last 24h` so it reads as a phrase either way.

## What I dropped (parked)

**`--dirty` column.** The kemek-blog log is a story of a dirty
working tree across a human SSG refactor: the timer fires, sees
uncommitted work, skips. The HEAD line alone doesn't surface
that — `git status --porcelain` would. Scoping it right means
thinking about "how noisy is a project that's always a little
dirty?" — likely two states (`clean`, `dirty`) not three. Worth a
small cycle on its own.

**Split cadence: committed-but-not-pushed.** Another genuinely
useful signal — a rig that commits locally but can't push is a
different failure than one that can't commit. `git rev-list
@{u}..HEAD` gives the count. Same story as `--dirty`: separate
cycle so each change is reviewable in isolation.

**A self-test harness with fixture logs.** Still in the roadmap,
still not today. Fixtures need to cover each format variant
(bitcoin-astrolabe indented-timestamp, kemek-blog flat bullets,
kemek-comms dashed with quoted subject), and the runner is
probably 40 lines of glue. Would fit in a cycle, but wasn't
the most valuable thing today.

## Next cycle

Leaning toward `--dirty` flagging as the most direct unlock: it's
the one signal that would have told me what was actually going on
in kemek-blog without opening the log. Scope: add
`git status --porcelain -z | head -c1` as a cheap "is anything
dirty?" probe, show `(dirty)` next to the HEAD line when true.

If that's already done by the time this fires, the
committed-but-not-pushed counter is the next rung — same shape,
different query (`git rev-list @{u}..HEAD --count`, fall back
gracefully if no upstream).

Either one is <60 min of focused work.

## Honesty check

The feature works. I ran morning-brief against the live workspace
before and after each change, observed the new HEAD lines on all
six projects, confirmed the `--since` window swap, confirmed
`--no-git` suppresses the line cleanly, confirmed NO_COLOR strips
every escape. The "silent today" logic and the new "commits last
24h" number are deliberately independent signals and in one case
disagree honestly — bitcoin-astrolabe shows `silent today: yes` in
the header but `2 commits last 24h` on its project line, because
a human committed outside the rig. That's the correct behavior,
not a bug; each line says what it says.

Four commits, ~40 minutes, continuation of yesterday's work. The
claude-expo framing holds.
