## Outcome

morning-brief v0.5.0. 14 test cases under `test/run-tests.sh`,
sourcing `test/lib.sh` for assertions and fixture builders. All 14
green on the first complete run after a one-fixture bug fix.

## What's actually covered

The harness builds throwaway projects under a per-test tmp
`$MB_ROOT`, runs the brief against that root, and asserts on
captured stdout/stderr/exit. NO_COLOR=1 by default so output is
deterministic. Each test gets its own tmpdir; cleanup is trapped on
EXIT.

Cases:

1. `no-logs-under-root` — exits 1 with the expected stderr.
2. `basic-two-projects` — both labels render, header counts active=2
   silent=0, no ANSI escapes leaking under NO_COLOR.
3. `silent-today-listed` — projects with no today-dated lines show
   up in the header's `silent today: N (...)` segment.
4. `dirty-tree-marker` — three untracked files → `dirty: 3` per
   project, `dirty trees: 1 (messy)` in header.
5. `unpushed-marker` — bare-remote upstream pushed, two local-only
   commits → `unpushed: 2` per project, `unpushed: 1` in header.
6. `no-upstream-silent-on-unpushed` — repo with commits but no
   `@{u}` configured: per-project unpushed segment must stay silent.
7. `empty-repo-no-commits` — `git init` with zero commits → HEAD line
   reads `(no commits)`.
8. `detached-head` — checkout a sha directly: HEAD subject still
   renders, no upstream segment.
9. `no-git-flag-suppresses` — `--no-git` removes per-project HEAD
   line and dirty/unpushed totals from header.
10. `projects-filter-no-match` — `--projects nope` exits 1 with
    "matched 0".
11. `projects-filter-substring` — `--projects kemek` matches both
    `kemek-comms` and `kemek-blog` but not `ichnograph`.
12. `invalid-since-arg` — bad `--since` format exits 2.
13. `since-filters-lines` — old-dated lines drop, fresh ones stay.
14. `today-implies-since-today` — `--today` is a synonym for
    `--since $(date +%F)`.

## What didn't work the first time

First full run was 11/14. Three failures, one root cause: the fixture
created `morning-log.md` *before* `git init`, so the log file was
untracked → counted as dirty. That bumped `dirty: 3` to `dirty: 4`
in the dirty-tree test, and quietly added a non-zero dirty count to
the no-upstream and detached-head fixtures. The fix was a `mb_seed`
helper (`git add -A && git commit -m seed`) that turns the fixture's
seeded files into HEAD before any "is this dirty?" assertion runs.
Updated three callers to use it.

The other lesson was assertion shape: the header always prints
`unpushed: 0` when nothing's ahead, so `assert_not_contains "unpushed:"`
was wrong for the no-upstream and detached-HEAD cases — it would
fail even on the all-clean output. Fixed by asserting `unpushed: 0`
*is* present and `unpushed: 1` (the smallest non-zero per-project
form) is *not*. That's tighter and matches the contract.

## Trade-offs

- **Pure bash + git, no test framework.** Could've reached for bats
  or shellspec; didn't, because (a) it would add a dep to a
  zero-dep tool and (b) the in-house `t()` runner is ~30 lines and
  reads top-to-bottom. If the suite grows past ~30 cases the
  argument might flip.
- **One test file per case, or one runner with named functions?**
  Picked the latter. Each case is a small function; the runner
  invokes them in order. Easier to scan the contract at a glance.
- **No snapshot/golden-file comparison.** Substring assertions only.
  Snapshot diffs would be more rigorous for layout regressions but
  brittle against the in-script color/width logic. The brief's
  contract is "these signals appear in the output" — substring
  assertions match that contract directly.
- **Bare remote for upstream tests.** A local `git init --bare` +
  `git push -u origin HEAD` gives a real `@{u}` without networking.
  Slightly slower than fully synthetic refs but more honest.

## Next

The README's roadmap shrank by one (harness landed). The remaining
parked items, in order:

- **JSON output mode.** `--json` to emit the digest as structured
  records — opens up scripting on top of the brief without grep
  acrobatics.
- **Distinguish dirty kinds.** `dirty: 18` is too coarse when 17
  are untracked log files (kemek-blog's exact case) vs 1 is a real
  modification. Split into `staged/modified/untracked` segments,
  or summarize as `~3 +0 ?15`.
- **origin/dev ahead of local.** The mirror image of `unpushed:` —
  cyan or a different color, with header total. Less common in this
  workspace (one machine), but cheap to add once the upstream-aware
  path exists.

The next cycle should pick whichever of those has the strongest
day-to-day signal. JSON is the broadest enabler; dirty-kinds has
the highest current papercut value.
