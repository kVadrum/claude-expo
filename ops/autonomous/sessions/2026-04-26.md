## What I shipped

morning-brief v0.4.0 — the unpushed-commit marker. Closes the
last visibility gap from yesterday's notes: a rig that commits
cleanly but never pushes was previously invisible to the brief.
`dev` ahead of `origin/dev` could pile up indefinitely while
every other signal looked fine.

Two layers, mirroring the dirty-tree shape from v0.3.0:

1. **Per-project `HEAD` line** gains a cyan `· unpushed: N ·`
   segment when `git rev-list @{u}..HEAD --count` is non-zero.
   Suppressed on a clean state and silently skipped when no
   upstream is configured (lib/, fresh repo, detached HEAD) —
   that's a different shape of "not pushed" and not the one this
   marker is trying to surface.

2. **Header summary** gains `unpushed: K (proj1, ...)` in cyan
   between `silent today` and `dirty trees`. Walks `all_logs` so a
   `--projects` drill-down on one rig still surfaces unpushed
   commits elsewhere — same survive-the-filter discipline as
   `silent_today` and `dirty_total`.

Color choice: cyan keeps the palette legible. Yellow already means
"waiting on a human" (skipped tail, dirty tree); green is "shipped
to dev"; red is "broken". Unpushed is none of those — it's
"committed but not yet on origin", a different shape of incomplete
shipping. Cyan reads as "informational, action-owed" without the
alarm of red or the human-gated tone of yellow.

## Honest dogfood moment

claude-expo itself was 17 commits ahead of origin/dev when this
cycle started. The first run of the new feature on the very repo
that built it surfaced exactly that — `unpushed: 17` on the
claude-expo line. Used it as the worked-example in the README too
(replacing a less-interesting bitcoin-astrolabe example block).
That `unpushed: 17` is also the shape of an autonomous rig that
the guardrails correctly keep on `dev` and never push: human
review gates the push, so unpushed commits accumulate until
kVadrum runs `push` interactively. The marker just makes that
visible at morning-review time.

Single git walk for both `dirty_total` and `unpushed_total` — they
live in the same project loop now, so the brief doesn't shell out
twice per project. Cheap to run is still a goal.

## What I dropped / didn't do

- **No JSON output mode.** Tempting as a follow-on but out of
  scope for one cycle. Roadmap'd in the README.
- **No symmetric `behind: N` marker** (origin ahead of local).
  Less common in this workspace but cheap to add once the
  upstream-aware path exists. Logged as roadmap.
- **No self-test harness.** Every cycle so far has tested by
  running against the live workspace, which is good for catching
  format-drift but bad for catching regressions in unusual states
  (no upstream, detached HEAD, empty repo). Still on the roadmap;
  still not the most useful cycle to spend.

## Trade-offs

- Walking `all_logs` for both dirty and unpushed makes a `--no-git`
  short-circuit valuable for a workspace with many projects. Kept
  the existing `--no-git` flag honoring both segments without
  needing a second flag.
- The "no upstream → silent skip" choice means a fresh repo with
  no remote shows neither `unpushed` nor `(no upstream)`. Could
  argue for a "not configured" hint, but it'd add noise on every
  lib/ entry. Silent felt right for the morning-review use case;
  noisy would make people tune the segment out.
- Cyan in the header always renders (even at 0) to keep the
  visual rhythm with `dirty trees: 0`. Could have hidden the
  segment when zero but the header would then have varying
  segment counts day-to-day — harder to scan.

## Next cycle

The README's roadmap, in order:

1. **Self-test harness with fixture logs.** Format changes have
   so far been caught by eyeballing the live brief, which won't
   scale and won't catch corner cases (no upstream, empty repo,
   `git` missing). Cheap to start: a `test/fixtures/` tree with
   five mock projects in five states, a smoke runner, and a few
   `assert grep -F` style checks.
2. **JSON output mode** for piping. Once the brief is stable,
   `--json` would let it feed a status bar or another rig. Not
   urgent.
3. **Distinguish dirty kinds** (modified-tracked vs untracked vs
   staged) when the count alone is too coarse to triage from.
   Would matter most in the kemek-blog "dirty: 18" case.

If the next cycle wants a clean break from morning-brief, the
workspace has another candidate: a small `concierge-brief` that
does for the concierge timers/units what morning-brief does for
morning-logs — armed-but-never-fired, last-fire timestamp, exit
status. Less ground to cover than another morning-brief feature
and a parallel idiom rather than a duplicated one.
